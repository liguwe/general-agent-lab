# submodule 工作流说明

`general-agent-lab` 是主工程，`codex/` 和 `opencode/` 是源码阅读上下文。两个目录用 Git submodule 固定到明确 commit，主工程只提交研究文档、原型、README 索引和 submodule 指针。

## 仓库关系

- `codex/` 指向 `git@github.com:liguwe/codex.git`，跟踪 `main`。
- `opencode/` 指向 `git@github.com:liguwe/opencode.git`，跟踪 `dev`。
- 两个 submodule 的 `origin` 是自己的 fork，用来保存少量贴近源码的说明文档。
- 两个 submodule 的 `upstream` 是原项目，用来同步官方更新：
  - `codex`: `https://github.com/openai/codex.git`
  - `opencode`: `https://github.com/anomalyco/opencode.git`

长期研究笔记优先放主工程 `docs/`。只有贴近源码仓库自身的说明文档，才提交到对应 fork。

## 初始化

第一次 clone 主工程时直接拉取 submodule：

```bash
git clone --recurse-submodules git@github.com:liguwe/general-agent-lab.git
```

如果已经 clone 了主工程，再初始化 submodule：

```bash
git submodule update --init --recursive
```

拉取主工程记录的 submodule 版本：

```bash
git submodule update --recursive
```

## 更新源码样本

更新到 fork 远端分支的最新 commit：

```bash
git submodule update --remote codex
git submodule update --remote opencode
```

更新后回到主工程提交 submodule 指针：

```bash
git status --short
git add codex opencode
git commit -m "chore(submodule): 更新源码样本指针"
```

## 同步上游

在对应 submodule 里拉取 upstream：

```bash
cd codex
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```

```bash
cd opencode
git fetch upstream
git checkout dev
git merge upstream/dev
git push origin dev
```

同步完成后回到主工程，提交新的 submodule 指针。

## 提交说明文档

如果说明文档贴近源码仓库自身，可以提交到对应 fork：

```bash
cd codex
git add README.zh.md AGENTS.zh.md CHANGELOG.zh.md SECURITY.zh.md
git commit -m "docs(codex): 更新中文说明文档"
git push origin main
```

然后回到主工程记录 submodule 指针：

```bash
cd ..
git add codex
git commit -m "chore(submodule): 更新 codex 指针"
```

`opencode/` 同理，但默认分支是 `dev`。

## 检查

```bash
git submodule status
git -C codex remote -v
git -C opencode remote -v
```

主工程的 `git status --short` 只应该出现 `.gitmodules`、`.gitignore`、README/docs、以及 `codex` / `opencode` gitlink 变化。
