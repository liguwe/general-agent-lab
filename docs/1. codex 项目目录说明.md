

这篇先回答三个问题：

- `codex/` 顶层每个目录和配置文件大致干什么。
- 现在的 Codex CLI 是怎么从命令跑起来的。
- 后续读源码应该优先看哪些 crate 和文件。

这里的粒度是“项目目录与关键入口文件”，不是递归解释每一个 Rust 源码文件。`codex-rs/core/src`、`tui/src` 这种目录下面文件很多，后续应该按主题继续拆，比如 Agent Loop、Tool、Approval、Sandbox、Thread。

## 1. 一句话结构

`codex` 这个仓库现在的核心是一个 Rust workspace：

```text
codex-cli/bin/codex.js
  -> 选择当前平台的 native codex 二进制
  -> codex-rs/cli/src/main.rs
  -> 无 subcommand 时进入 codex_tui::run_main
  -> codex exec / review 时进入 codex_exec::run_main
  -> tui / exec 继续通过 app-server / core 驱动 thread、turn、tool、sandbox、model 调用
```

所以读源码时不要被 npm 包迷惑。`codex-cli` 是安装和分发入口，真正的 CLI 主体在 `codex-rs`。

## 3. `codex-cli`：npm 安装入口

`codex-cli/package.json` 暴露了 npm bin：

```json
{

    "codex": "bin/codex.js"
  }
}
```

关键文件：

| 路径 | 作用 |
| --- | --- |
| `codex-cli/bin/codex.js` | Node wrapper。根据 `process.platform` 和 `process.arch` 选择 target triple，再找到对应平台包里的 native `codex` 二进制，然后 `spawn(binaryPath, process.argv.slice(2))`。 |
| `codex-cli/bin/rg` | 随包附带/辅助的 ripgrep 入口。 |

| `codex-cli/scripts/install_native_deps.py` | 安装 native 依赖。 |
| `codex-cli/scripts/run_in_container.sh` | 容器内运行辅助脚本。 |

## 5. `codex` 命令怎么跑起来

### 5.1. npm / brew 安装后实际执行 native binary

用户安装的是 `@openai/codex` 或 Homebrew cask。npm 形态下，shell 中的 `codex` 先进入：

```text
codex-cli/bin/codex.js
```

这个脚本做三件事：

1. 根据平台和架构映射 target triple，例如 macOS arm64 -> `aarch64-apple-darwin`。
2. 找到对应平台 npm optional package 或本地 `vendor` 里的 native binary。
3. 用 `spawn` 执行 native `codex`，并把原始参数转发过去。

### 5.2. Rust 总入口解析 subcommand

native binary 对应：

```text
codex-rs/cli/src/main.rs
```

`MultitoolCli` 用 clap 定义了顶层命令：

- 无 subcommand：进入交互式 TUI。
- `exec` / `review`：进入非交互运行。
- `login` / `logout`：认证管理。
- `mcp` / `mcp-server`：MCP 管理和 server 模式。
- `plugin`：插件管理。
- `sandbox` / `debug` / `execpolicy`：调试、沙箱和执行策略。
- `resume` / `fork`：恢复或 fork 历史 session。
- `app-server` / `remote-control` / `exec-server`：内部服务和远程控制能力。

核心分发在 `cli_main`：

```text
match subcommand {
  None => run_interactive_tui(...)
  Some(Subcommand::Exec(...)) => codex_exec::run_main(...)
  Some(Subcommand::Review(...)) => codex_exec::run_main(...)
  ...
}
```

### 5.3. 交互模式进入 TUI

无 subcommand 的 `codex` 会进入：

```text
codex-rs/tui/src/main.rs
codex-rs/tui/src/lib.rs
```

`tui/src/main.rs` 本身很薄：解析 TUI CLI 参数，调用 `codex_tui::run_main`，退出时打印 token usage 和 resume hint。

TUI 下面的重点目录：

| 路径 | 作用 |
| --- | --- |
| `tui/src/app.rs`、`tui/src/app/` | TUI app 生命周期、事件分发、thread 路由、session 状态。 |
| `tui/src/chatwidget.rs`、`tui/src/chatwidget/` | 主要聊天界面、输入、turn 生命周期、tool 请求、流式渲染。 |
| `tui/src/bottom_pane/` | 底部输入区、弹窗、approval overlay、slash command、file search popup。 |
| `tui/src/history_cell/` | 历史消息、命令执行、MCP、approval 等渲染 cell。 |
| `tui/src/exec_cell/` | 命令执行 cell 模型和渲染。 |
| `tui/src/diff_model.rs`、`tui/src/diff_render.rs` | diff 展示。 |
| `tui/src/app_server_session.rs` | TUI 和 app server session 的桥。 |

### 5.4. 非交互模式进入 exec

`codex exec` 的入口有两层：

```text
codex-rs/cli/src/main.rs
  -> codex_exec::run_main(...)

codex-rs/exec/src/main.rs
  -> 单独 codex-exec binary 的入口
```

`exec/src/lib.rs` 是重点。它会：

1. 解析 prompt / stdin / review 参数。
2. 加载配置、登录状态、model provider、sandbox/approval 选项。
3. 启动 `InProcessAppServerClient`。
4. 发起 `ThreadStart` 或 `ThreadResume`。
5. 发起 `TurnStart`。
6. 用 `EventProcessor` 把 app server 的 notification 转成 human output 或 JSONL output。

这说明 Codex 的非交互 CLI 不是直接在 `exec` 里手写 Agent Loop，而是复用 app-server/thread/turn 这套协议。

### 5.5. app-server 承接 thread / turn / 请求处理

重点路径：

```text
codex-rs/app-server/src
codex-rs/app-server-protocol/src
```

`app-server-protocol` 定义 JSON-RPC 协议和数据结构，比如：

- `ThreadStart`
- `ThreadResume`
- `TurnStart`
- server notification
- thread item

`app-server/src/request_processors/` 是请求处理器集合。后续拆 Agent Loop 时，应该重点看：

| 路径 | 作用 |
| --- | --- |
| `request_processors/thread_processor.rs` | thread 相关请求处理。 |
| `request_processors/turn_processor.rs` | turn 启动和执行。 |
| `thread_state.rs` | thread runtime 状态。 |
| `message_processor.rs` | app-server 消息处理。 |
| `dynamic_tools.rs` | 动态工具。 |
| `command_exec.rs` | 命令执行。 |

## 6. 当前源码阅读顺序

第一轮不要从 `core/src` 里随机钻。建议按运行链路读：

1. `codex-cli/bin/codex.js`
2. `codex-rs/cli/src/main.rs`
3. `codex-rs/tui/src/main.rs`
4. `codex-rs/exec/src/lib.rs`
5. `codex-rs/app-server-protocol/src/protocol/mod.rs`
6. `codex-rs/app-server/src/request_processors/turn_processor.rs`
7. `codex-rs/core/src/codex_thread.rs`
8. `codex-rs/core/src/tools/`
9. `codex-rs/core/src/exec_policy.rs`
10. `codex-rs/core/src/rollout.rs`

读完这条链路，基本就能回答第一阶段的问题：一个 Code Agent CLI 至少需要哪些模块。

## 7. 对 mini-code-agent-cli 的映射

Codex 的结构很大，但对 `mini-code-agent-cli` 第一版可以只取骨架：

| Codex 模块 | mini 原型里的最小对应 |
| --- | --- |
| `codex-cli/bin/codex.js` | 暂时不做，先直接跑本地 CLI。 |
| `cli` | `mini-agent run` / `mini-agent exec` 命令分发。 |
| `tui` | 暂时不用全屏 TUI，先做普通 stdout event stream。 |
| `exec` | 非交互 prompt 执行入口。 |
| `app-server-protocol` | 最小事件协议：`ThreadStart`、`TurnStart`、`ToolCall`、`TurnDone`。 |
| `core` | Agent Loop、上下文、工具注册、权限判断。 |
| `tools` | `shell`、`read_file`、`apply_patch` 三个工具。 |
| `execpolicy` / `sandboxing` | 先做策略判断，不做真实 OS sandbox。 |
| `rollout` / `thread-store` | JSONL session log。 |

这个映射是后续原型的判断标准：每读完 Codex 一个模块，必须问它在 mini agent 里对应什么。如果对应不上，就先不展开。
