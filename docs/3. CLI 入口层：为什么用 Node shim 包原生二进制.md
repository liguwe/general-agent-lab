# 3. CLI 入口层：为什么用 Node shim 包原生二进制

## 问题

Codex 和 opencode 的底层核心分别是 Rust 和 Go，但用户安装的入口 `npm i -g @openai/codex` 或 `npm i -g opencode` 拿到的是一个 Node.js 脚本。为什么不只发原生二进制？

## 实际分层

两个项目的真实架构一致：

| 层 | Codex | opencode |
|---|---|---|
| 入口 shim | `bin/codex.js`（~240 行, Node） | `packages/opencode/bin/opencode`（~200 行, Node） |
| 核心二进制 | Rust 编译产物 `codex` | Go 编译产物 `opencode` |
| 分发渠道 | npm（`@openai/codex`） | npm（`opencode`） |

shim 的内容很简单：检测平台 → 从 `node_modules` 里找到对应的原生二进制 → `child_process.spawn` 启动 → 透传 stdin/stdout/stderr → 转发信号。

## shim 承担的四个职责

### 1. 多平台分发

npm 的 optional dependencies 机制让一个包名覆盖 6 种平台二进制：

```
@openai/codex
├── @openai/codex-darwin-arm64    (Apple Silicon)
├── @openai/codex-darwin-x64      (Intel Mac)
├── @openai/codex-linux-x64       (x86 Linux)
├── @openai/codex-linux-arm64     (ARM Linux)
├── @openai/codex-win32-x64       (Windows)
└── @openai/codex-win32-arm64     (Windows ARM)
```

用户 `npm i -g @openai/codex` 只拉自己平台的包。shim 运行时通过 `process.platform` + `process.arch` 解析出对应的二进制路径。

opencode 更进一步，在 x64 Linux 上还区分了 **AVX2 / baseline / musl / glibc** 四种变体，运行时检测 CPU 特性选最优版本。

### 2. 信号转发与退出码透传

直接跑原生二进制也能用，但 Ctrl-C 处理会不干净。shim 做的事情：

```js
// 转发终止信号
["SIGINT", "SIGTERM", "SIGHUP"].forEach(sig => {
  process.on(sig, () => child.kill(sig))
})

// 子进程退出后，镜像退出原因
child.on("exit", (code, signal) => {
  if (signal) process.kill(process.pid, signal)
  else process.exit(code ?? 1)
})
```

这让脚本化的使用场景（CI、管道、supervisor）能拿到正确的退出码和信号语义。

### 3. 包管理器感知

shim 检测安装时用的是 npm 还是 bun，设置对应的环境变量：

```
CODEX_MANAGED_BY_NPM=1
CODEX_MANAGED_BY_BUN=1
CODEX_MANAGED_PACKAGE_ROOT=/path/to/global/node_modules/@openai/codex
```

原生二进制可以据此调整更新提示文案、日志路径等行为。

### 4. PATH 注入

shim 会把原生二进制所在目录前插到 `PATH`，这样二进制在执行过程中 spawn 的其他进程（比如 git、ripgrep）能优先找到同包 vendor 里的工具链。

## 为什么不走 brew / 直接发二进制

两个项目都同时提供 brew 安装，但 npm 作为**主渠道**的原因：

| 因素 | 说明 |
|---|---|
| 开发者用户群 | 目标用户都已有 Node/npm，`npm i -g` 零心智负担 |
| 自动更新 | `npm update -g` 或 `npm i -g @latest`，不用自建更新管道 |
| optional deps | 天然解决多平台，不需要自己搞 S3 + checksum 下载脚本 |
| 生态整合 | 自动获得 VS Code / JetBrains 插件的 npm 依赖安装能力 |

Codex 的 `install` 脚本（`codex/install`）确实提供了不依赖 npm 的 fallback 下载方案，但那只是兜底，主渠道依然是 npm。

## 对 mini-code-agent-cli 的启示

如果最终也走 **原生二进制 + JS shim** 模式：
- shim 负责分发、信号、环境检测，代码量 ~200 行
- 原生二进制负责重活：LLM 调用、代码执行、权限控制、TUI

如果做纯 JS/TS 版本，就不需要这层分离。opencode 分出来的原因是它的 TUI（Solid.js + OpenTUI）和 LLM 客户端编译成 Go 二进制性能更好、体积更小、不依赖 Node 运行时。Codex 用 Rust 则主要是为了安全性和编译后单二进制部署的简洁性。

当前阶段做原型时可以先用纯 Node，等验证了交互模式、确认哪些路径对性能敏感之后，再决定是否需要拆出原生层。
